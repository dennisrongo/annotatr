How It Works

All three hotkeys use the existing hotkey infrastructure implemented in Features #56, #57, #58:

  1. App Startup: MiniPanel.tsx calls register_hotkeys() with default settings
  2. Registration: Backend iterates through all hotkeys in config and registers them with OS
  3. Trigger: When user presses Ctrl+Shift+F/T/H, the global shortcut is triggered
  4. Handler: activate_tool_hotkey() is called with the hotkey config key
  5. Conversion: convert_hotkey_tool_name() converts "freehandTool" → "freehand"
  6. Activation: Overlay is shown, drawing mode enabled, tool-selected event emitted with "freehand"
  7. UI Update: Overlay receives tool-selected event and sets current tool to ToolType.FREEHAND

Fix Details

Problem

The hotkey configuration uses keys like "freehandTool", "textTool", "highlighterTool" but the ToolType enum uses values like "freehand", "text", "highlighter". Without conversion, the tool-selected event would emit "freehandTool" which doesn't match any ToolType value, causing the tool selection to fail silently.

Solution

Added convert_hotkey_tool_name() function in src-tauri/src/lib.rs:

/// Convert hotkey config key to ToolType value
/// Example: "freehandTool" -> "freehand", "textTool" -> "text"
fn convert_hotkey_tool_name(hotkey_key: &str) -> String {
    match hotkey_key {
        "arrowTool" => "arrow".to_string(),
        "circleTool" => "circle".to_string(),
        "boxTool" => "box".to_string(),
        "freehandTool" => "freehand".to_string(),
        "highlighterTool" => "highlighter".to_string(),
        "textTool" => "text".to_string(),
        _ => hotkey_key.to_string(), // Return as-is if no match
    }
}

This function is called in activate_tool_hotkey() before emitting the tool-selected event, ensuring the overlay receives a valid ToolType enum value.
