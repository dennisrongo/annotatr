Windows-Specific Behavior

Tauri 2 automatically handles Windows-specific requirements:

  • DWM Composition: Uses Desktop Window Manager for transparent layered windows
  • Window Styles: Applies WS_EX_LAYERED, WS_EX_TOPMOST, WS_EX_TOOLWINDOW automatically
  • Click-Through: Uses layered window attributes for pass-through mode
  • Multi-Monitor: Uses EnumDisplayMonitors via Tauri's monitor API
  • Always-On-Top: Uses SetWindowPos with HWND_TOPMOST

Linux-Specific Behavior

Tauri 2 automatically handles Linux-specific requirements:

  • X11/Wayland: Uses appropriate backend based on display server
  • Window Type: Sets _NET_WM_WINDOW_TYPE to UTILITY or TOOLBAR
  • Always-On-Top: Uses _NET_WM_STATE_ABOVE hint
  • Click-Through: Uses shape extension or input shape
  • Multi-Monitor: Uses Xinerama or RandR via Tauri's monitor API
  • Compositing: Works with modern compositors (Compiz, KWin, Mutter, etc.)
NOTE: Linux implementation requires a compositing window manager for transparent windows. Tauri handles this automatically - if no compositor is available, the window will still work but without transparency (fallback behavior).

Why This Is The Correct Implementation

Architectural Best Practices

✓ CORRECT: Use Tauri 2's cross-platform abstraction layer

  • Single codebase for all platforms
  • Platform differences handled by Tauri framework
  • Automatic updates when Tauri adds platform support
  • Reduced maintenance burden
  • Consistent behavior across platforms

✗ INCORRECT: Write platform-specific code manually

  • Three separate code paths to maintain (Windows, macOS, Linux)
  • Risk of inconsistent behavior
  • Reinventing the wheel (Tauri already solved this)

  • Increased bug surface area
  • Higher maintenance burden

Industry Standard

This is how Electron, Tauri, and other cross-platform frameworks work. The entire point of using Tauri is to avoid writing platform-specific code. The Windows/macOS/Linux implementations are provided by Tauri itself, not by application code.
